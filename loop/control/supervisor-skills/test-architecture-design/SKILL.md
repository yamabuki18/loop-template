# テストアーキテクチャ設計によるテストケース導出

PRD・Design Doc・実装コードから、**論理的機能構造**でテストケースの単位を識別し、**重篤度**で優先度を決め、**テストサイズ**で実装レベル（配置）を決める。これにより「何を・どの深さで・どの層のテストとして」書くべきかを体系的に導出する。

出典: freee ren氏「テストアーキテクチャ設計で実現する高品質で高スピードな開発の実践」(2025)。
論理的機能構造は湯本剛氏のゆもつよメソッド、テストサイズは『Googleのソフトウェアエンジニアリング』に基づく。

## 全体フロー

```
入力: PRD / Design Doc / 実装コード / ソフトウェアアーキテクチャ情報
  │
  1. 論理的機能構造を用いた機能リストの作成
  │    ドメイン分割 → フィーチャー特定 → 論理的機能構造アイテムに分解
  │    （論理的機能構造アイテム ≒ テストケースの単位）
  │
  2. 重篤度に基づいたテストケースの選定
  │    フィーチャー単位のリスク分析 → アイテム単位の重篤度分類
  │
  3. テストサイズ分類による実装レベルの決定
  │    アーキテクチャ対応からテストサイズを特定 → 自動化方針を決定
  │
  4. (TDDモード) テスト設計表に沿って Red-Green-Refactor
  │
出力: テスト設計表（アイテム × 重篤度 × テストサイズ × 実装レベル）+ テストコード
```

## Step 1. 論理的機能構造を用いた機能リストの作成

PRD・Design Doc・実装コードを読み、次の3段階で階層的な機能リストを作る。

1. **ドメイン分割**: テスト対象システムを業務ドメインごとに分割する（例: 振込、残高管理、通知）。
2. **フィーチャー特定**: ドメイン内で「ユーザー視点で観測可能な振る舞い」ごとにフィーチャーを特定する。
   フィーチャーとは、ユーザーが使う際にユーザー自身で認識できる機能のこと（例: 「振込を実行する」）。
3. **論理的機能構造アイテムの特定**: 各フィーチャーを下表の7要素で分析し、具体的な処理単位に分解する。
   この**論理的機能構造アイテムがテストケースの単位**になる。

### 論理的機能構造の7要素

| 要素 | 定義 | 例 |
|---|---|---|
| 入力調整 | 外部からの入力を受け付け、内部処理用に整える処理 | リクエストのパース、入力値の正規化 |
| 出力調整 | 内部の処理結果を外部向けに整えて出力する処理 | レスポンス整形、画面表示用フォーマット |
| 変換 | 入力を別の形・値に変換するコアな処理ロジック | 振込金額計算、税額計算 |
| 貯蔵 | データの保存・読み出し | DBへの永続化、キャッシュ |
| サポート | フィーチャー内部の補助処理（自フィーチャー内呼び出し） | バリデーション、ステータス管理 |
| 相互作用 | 他フィーチャー・他サービスとの連携（外部呼び出し） | 他プロダクトへの通知、外部API呼び出し |
| 外部マネジメント | 外部からの管理・設定に関わる処理 | 権限設定、機能フラグ、運用設定 |

アイテムは「処理名 / [要素名]」の形式で記述する（例: `振込金額計算 / [変換]`）。

**注意**: 仕様書に書かれた機能を写経するのではなく、「このフィーチャーを実現するためにシステム内部で何をしているか」を7要素をガイドに想定・分解する。仕様に明記されていない暗黙の処理（バリデーション、エラー時の出力調整など）の抜け漏れを防ぐのがこのモデルの目的。

## Step 2. 重篤度に基づいたテストケースの選定

重篤度とは「事象のひどさ」＝ユーザーのもとで発生してはいけない度合い。

| 重篤度 | 定義 |
|---|---|
| Critical | サービスが全面的に使えない |
| Major | 主要な機能が動かない、動作不正のため役に立たない |
| Normal | 機能が動かない・動作不正があるが、サービスは継続利用できる |
| Minor | 仕様通りではないが、サービスを利用した結果には影響がない |

### 選定手順

1. フィーチャー単位でリスク分析を行う（このフィーチャーが壊れたら何が起きるか）。
2. 高リスクなフィーチャーについて、論理的機能構造アイテム単位で「壊れたときの事象」を想定し重篤度を付与する。
3. **重篤度が高い（Critical / Major）アイテムをリグレッションテストスイート（＝変更のたびに必ず走らせるテスト）に含める**。

### TDD適用時の解釈

- **Critical / Major**: テストファーストで実装する。正常系に加え、境界値・異常系まで網羅的にテストケースを書く。
- **Normal**: 正常系＋代表的な異常系のテストを書く。
- **Minor**: 最小限のテスト、またはテスト省略を許容する（工数対効果で判断）。

**重篤度は自動判定する**: ユーザーへの確認は挟まず、AIがすべてのアイテムの重篤度を判定して進める。ただしテスト設計表の備考欄等で判断根拠（壊れたときに何が起きるか）を1行で示し、ユーザーが後から見直せる状態にする。なお元手法では判断基準をSWE・QA・PM・デザイナーの合議で決めており、判定に迷う場合は「金銭・データ損失や信頼毀損に直結するか（Critical）／主要機能が役に立たなくなるか（Major）／継続利用は可能か（Normal）／結果に影響がないか（Minor）」の順に問いを立てて分類する。

## Step 3. テストサイズ分類による実装レベルの決定

テストサイズは「テストがどう実行され、何を許可され、どれだけのリソースを消費するか」で決まる（『Googleのソフトウェアエンジニアリング』）。

| サイズ | 制約 | 特徴 |
|---|---|---|
| Small | 単一プロセス内で完結。I/O・sleep・ブロッキング呼び出し・ネットワーク禁止 | 高速・決定的。CIで常時実行できる |
| Medium | 単一マシン内で完結。localhostへのネットワーク呼び出しは可（実DB・複数プロセス可） | 統合的な検証が可能だがやや遅く、flakyになりうる |
| Large | 制約なし。複数マシン・実外部サービスにまたがる | 忠実性は最高だが遅く不安定。最小限に絞る |

### アーキテクチャ対応によるテストサイズの特定

論理的機能構造とソフトウェアアーキテクチャには対応関係があり、「システム分割の見方」を介してテストサイズを特定できる。以下はクリーンアーキテクチャ系の構成を想定した目安（対象システムの実アーキテクチャに合わせて調整すること）:

| 論理的機能構造 | 対応するアーキテクチャ要素 | テストサイズの目安 |
|---|---|---|
| 変換 | domain / entity 層のロジック | Small |
| サポート（バリデーション等） | domain / adapter 層 | Small |
| 入力調整・出力調整 | adapter / controller 層 | Small（モック併用）〜 Medium |
| 貯蔵 | repository + DB | Medium（localhost の実DBを使用） |
| 相互作用（自社管理のサービス連携） | entrypoint（API / batch / worker）単位の統合処理 | Medium |
| 相互作用（外部管理のサービス連携） | 外部サービスとの実結合 | Large（または手動テスト） |
| 外部マネジメント | 設定・インフラ層 | ケースバイケース（多くは Medium 以上） |

判断の原則:
- コンポーネント内で完結するテストは、そのコンポーネントの**責務と切り分け方**からサイズを特定できる。
- コンポーネントを跨ぐ統合処理は、**外部サービスとの連携有無と、そのサービスの管理主体**でサイズが決まる。自社管理ならlocalhost/テスト環境で代替でき Medium、外部管理なら Large。
- 同じ「相互作用」でも連携先次第で Medium にも Large にもなる。要素名だけで機械的に決めない。

### 自動化方針

- **Small / Medium** → 自動テストとして実装し、CIパイプラインに統合する。
- **Large** → 自動化コストと保守コストを天秤にかける。UX検証や外部サービスとの実結合など、自動化が割に合わないものは手動テストとして残す。
- 高重篤度 × Small〜Medium のアイテムがCI化の最優先ターゲット。**安易なE2E自動化に逃げない**（E2Eの保守コストがスピードを殺す、というのがこの手法の出発点）。

## Step 4. TDDへの適用

テスト設計表が完成したら、以下の順でテストファースト開発を行う。

1. **実装順序**: 重篤度が高く、テストサイズが小さいアイテムから着手する（Critical×Small → Major×Small → Critical×Medium → …）。Smallテストで守られたコアロジックの上に統合を積む。
2. **Red**: アイテム1件につき、対応するテストレベル（対応表参照）で失敗するテストを書く。変換・サポートは Table Driven Test 等で入出力の網羅を重視する。
3. **Green → Refactor**: 実装しテストを通し、リファクタリングする。テストは内部実装ではなく処理のI/O（ブラックボックス）に対して書き、実装変更に耐えるようにする。
4. **統合レベル**: entrypoint（API / batch / worker）単位では「最長のハッピーパス」を少なくとも1本 Medium テストとして書く。細部の網羅は Small に寄せ、Medium で重複させない。
5. テストサイズを跨いで**同じ観点を重複して検証しない**。Smallで担保済みのロジックをMedium/Largeで再検証するのはコストの二重払い。

## 出力フォーマット: テスト設計表

以下のスキーマでMarkdown表を出力する。

| ドメイン | フィーチャー | 論理的機能構造アイテム | 重篤度 | テストサイズ | 実装レベル | リグレッション対象 | 備考 |
|---|---|---|---|---|---|---|---|
| 振込 | 振込を実行する | 振込金額計算 / [変換] | Critical | Small | Unit Test (domain) | ✅ | 境界値を網羅 |
| 振込 | 振込を実行する | 振込依頼提出 / [相互作用] | Critical | Medium | Integration Test | ✅ | 自社管理サービス連携 |
| 振込 | 振込を実行する | ステータス更新のバリデーション / [サポート] | Major | Small | Unit Test | ✅ | |
| 振込 | 振込を実行する | 関連プロダクトへのステータス更新 / [相互作用] | Major | Large | E2E / 手動 | ✅ | 外部管理サービス |
| 振込 | 振込を実行する | 振込成功に伴うメール通知 / [相互作用] | Normal | Medium | Integration Test | - | 振込自体には影響小 |

重篤度の判断例: 振込金額計算の誤りはユーザーに直接的な金銭的損失を与えるため Critical。メール通知の不正はユーザーの不便は生じるが振込行為自体には影響が少ないため Normal。

## 実行時の注意

- 入力ドキュメントが不足している場合（アーキテクチャ情報がない等）は、対応表のどの行が適用できないかを明示し、ユーザーに確認する。
- ドメイン・フィーチャーの粒度に迷ったら、まず粗く分割して全体を出し、ユーザーとすり合わせてから細分化する。
- この手法の狙いは「重大な事象の検知に集中した効果的なスイート」×「実行コストを最適化した効率的な自動テスト」の両立。テストケースを増やすことではなく、**書くべきテストと書かなくてよいテストを判別すること**に価値がある。

## ループ運用への写像（loop supervise / plan mode で使うとき）

このスキルは**監督の対話セッション専用**（supervise.sh が配布する）。自律ループの planner /
worker はこのスキルを持たない — 設計判断はここ（人間との対話）で行い、成果物だけをループに
渡す。plan mode で設計する場合、**テスト設計表を計画本文に含めて ExitPlanMode すること**。
計画は memory/plans/ に捕捉され、`loop handoff` 経由で planner が「その計画に忠実に」スライス
と契約テストを切る。設計表が計画に無ければ、この手法の判断はループに一切伝わらない。

設計表の各行は、ループの語彙へ次のように振り分けて計画に明記する:

| 設計表上の判定 | ループでの配置 |
|---|---|
| Critical / Major × Small〜Medium | `tests/` の契約テスト（F2P）候補 — planner が書き、gate が強制する |
| Normal | ワーカーの co-located 単体テスト（スライス brief に観点を書く） |
| Minor | 省略可（書かない判断を計画に残す） |
| Large（E2E / 実外部結合） | **gate に置かない** — 手動テスト or 監督専有スモーク。`frontend-humble-object` パック導入時は E2E スイート新設が L2/L3 でブロックされる前提で設計する |

- Medium テスト（localhost 実 DB 等)を契約テストにする場合は、gate 環境にその依存が
  用意されていることを先に確認する（無ければ Small へ寄せる）。
- リグレッション対象 ✅ の行は、landing 後は P2P（既存スイート）としてゲートに残り続ける。
  「変更のたびに必ず走らせる」はループでは gate がそのまま実現する。
