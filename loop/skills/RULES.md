# RULES — このプロジェクトの規約（助言層 L1）

> ワーカー/計画役のClaudeが従う**助言的**ルール。守らせたい度合いが上がったものだけ、
> 順にハーネス(フック=決定論ブロック)や構造テスト(破ると落ちる)へ「昇格」させる。
> 最初から全部を絶対ルールにしないこと（escalation ladder）。

## コーディング規約
- 命名: <例: 関数camelCase / 定数UPPER_SNAKE>
- スタイル: <例: フォーマッタに従う。手で整形しない>
- コメント: <例: 周囲のコードの密度に合わせる。自明なコメントは書かない>

## やってはいけないこと
- <例: 既存の公開APIのシグネチャを断りなく変えない>
- <例: 巨大な無関係リファクタを混ぜない。1論点=1コミット>
- `tests/` を編集しない（受け入れテストは監督の仕様＝合格基準。ハーネスがブロックする）

## コミット/作業の粒度
- 論理単位ごとにこまめにコミット（コミットは自動pushされ、gateが常に最新を見る）
- 完了したら `/work/STATUS` に1行サマリを追記し末尾を `DONE` にする

## テスト方針
- ワーカーは**全体スイートを回さない**（監督=gateが回す）。自分のsrc配下にco-locationの単体テストは書いてよい
- 合格基準は `tests/` の受け入れテスト。これを通すことがタスクの定義

## 昇格ログ（破られて昇格させたルール）
- <日付 / ルール / 昇格先(フック or 構造テスト)> — 例: 2026-06-15 tests/編集 → gate exit4 で強制
